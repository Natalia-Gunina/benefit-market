# Глава 3. Проектирование и разработка информационной системы «Benefit Market»

## 3.1. Архитектура приложения

### 3.1.1. Текущая архитектура

Система реализована как **монолитное full-stack веб-приложение** на базе фреймворка Next.js 16 с использованием паттерна App Router. Выбор монолитной архитектуры обусловлен масштабом MVP-версии продукта: единая кодовая база упрощает разработку, отладку и развёртывание на начальном этапе.

Приложение разделено на три логических слоя:

```
┌──────────────────────────────────────────────────┐
│            Клиент (SPA, React 19)                │
│   shadcn/ui · Zustand · TanStack React Query     │
└────────────────────┬─────────────────────────────┘
                     │ HTTPS / JSON
┌────────────────────▼─────────────────────────────┐
│          Серверный слой (Next.js 16)             │
│  API Routes (56 эндпоинтов) · Middleware (RBAC)  │
│  Сервисы бизнес-логики · SSR-рендеринг           │
└────────────────────┬─────────────────────────────┘
                     │ PostgreSQL Protocol
┌────────────────────▼─────────────────────────────┐
│        Уровень данных (Supabase / PostgreSQL)    │
│  18 таблиц · 37 RLS-политик · JWT Auth           │
└──────────────────────────────────────────────────┘
```

**Презентационный слой** — React-компоненты с разделением на серверные (загрузка данных) и клиентские (интерактивность). Управление серверным состоянием — TanStack React Query, клиентским — Zustand.

**Слой бизнес-логики** — сервисы (`order.service`, `analytics.service`, `accrual.service`), доменные правила (evaluator элигибильности), Zod-валидация входных данных.

**Слой данных** — PostgreSQL с Row-Level Security, обеспечивающей мультитенантную изоляцию на уровне СУБД без дополнительного кода в приложении.

### 3.1.2. Целевая архитектура

На этапе масштабирования планируется переход к **модульному монолиту** с выделением bounded contexts:

| Модуль | Ответственность | Критерий выделения |
|--------|----------------|-------------------|
| `benefits-core` | Каталог льгот, правила элигибильности | Высокая частота изменений бизнес-правил |
| `marketplace` | Провайдеры, офферинги, отзывы | Независимый цикл разработки |
| `billing` | Кошельки, начисления, списания | Требования к транзакционной целостности |
| `identity` | Аутентификация, RBAC, профили | Выделение в отдельный сервис при интеграции с SSO |
| `integration` | Коннекторы HRIS / 1С | Изоляция внешних зависимостей |

Переход планируется без изменения инфраструктуры: модули выделяются как пакеты внутри монорепозитория, взаимодействуют через типизированные интерфейсы. При необходимости горизонтального масштабирования модули могут быть вынесены в отдельные микросервисы.

### 3.1.3. Стек технологий и обоснование выбора

#### Серверная часть

| Технология | Версия | Обоснование |
|-----------|--------|------------|
| **Next.js** (App Router) | 16.1.6 | Гибридный рендеринг (SSR/CSR), встроенные API Routes устраняют необходимость отдельного бэкенда, Turbopack ускоряет сборку |
| **TypeScript** | 5.x (strict) | Сквозная типизация от БД до UI снижает количество runtime-ошибок; Zod-схемы обеспечивают валидацию на границах системы |
| **Supabase** | 2.97.0 | Управляемый PostgreSQL с встроенной аутентификацией (JWT), RLS-политиками и realtime-подписками; сокращает время вывода MVP |
| **PostgreSQL** | 15+ | Поддержка JSONB для гибких схем (настройки тенанта, условия элигибильности), RLS для мультитенантной изоляции |
| **Node.js** | 20 LTS | Долгосрочная поддержка, единый язык на клиенте и сервере |

#### Клиентская часть

| Технология | Версия | Обоснование |
|-----------|--------|------------|
| **React** | 19.2.3 | Server Components для SSR без гидратации, Concurrent-рендеринг для отзывчивости UI |
| **shadcn/ui + Radix UI** | 1.4.3 | Доступные (WCAG 2.1) headless-компоненты; копируются в проект — полный контроль над кодом |
| **Tailwind CSS** | 4.x | Utility-first подход, tree-shaking неиспользуемых стилей, минимальный CSS-бандл |
| **TanStack React Query** | 5.90.21 | Кеширование, фоновая ревалидация, оптимистичные обновления — решает проблему синхронизации серверного состояния |
| **Zustand** | 5.0.11 | Минимальный стор (< 1 КБ) с persist-middleware для корзины; проще Redux для масштаба проекта |
| **Recharts** | 3.7.0 | Декларативные React-графики для аналитических дашбордов |

#### Инфраструктура

| Технология | Обоснование |
|-----------|------------|
| **Docker** (multi-stage, Alpine) | Воспроизводимые сборки, standalone-выход Next.js (~50 МБ образ) |
| **Caddy** | Автоматический HTTPS (Let's Encrypt), HTTP/2, минимальная конфигурация |
| **GitHub Actions** | CI/CD в одной экосистеме с кодом; 2 пайплайна: тестирование и деплой |
| **Vitest** | Совместимость с Vite/TypeScript, скорость выполнения тестов |

### 3.1.4. Схема базы данных

База данных содержит 18 таблиц, 8 ENUM-типов и 37 RLS-политик. Ниже представлена ER-диаграмма основных сущностей:

```
                        ┌───────────┐
                        │  tenants  │
                        └─────┬─────┘
                         1:N  │
              ┌───────────────┼──────────────────┐
              ▼               ▼                  ▼
        ┌──────────┐   ┌───────────────┐  ┌──────────────┐
        │  users   │   │budget_policies│  │benefit_categ.│
        └────┬─────┘   └───────────────┘  └──────┬───────┘
             │                                    │ 1:N
     ┌───────┼────────┐                    ┌──────▼──────┐
     ▼       ▼        ▼                    │  benefits   │
┌────────┐┌──────┐┌───────┐               └──────┬──────┘
│wallets ││orders││empl.  │                      │ 1:N
└───┬────┘└──┬───┘│profile│               ┌──────▼──────────┐
    │ 1:N    │1:N └───────┘               │eligibility_rules│
┌───▼──────┐┌▼──────────┐                 └─────────────────┘
│pt_ledger ││order_items │
└──────────┘└────────────┘

        МАРКЕТПЛЕЙС:
┌───────────┐  1:N  ┌──────────────────┐  N:M  ┌─────────────────┐
│ providers ├──────▶│provider_offerings├──────▶│tenant_offerings │
└─────┬─────┘       └────────┬─────────┘       └─────────────────┘
      │ 1:N                  │ 1:N
┌─────▼────────┐      ┌─────▼───┐
│provider_users│      │ reviews │
└──────────────┘      └─────────┘
```

**Ключевые таблицы:**

| Группа | Таблицы | Назначение |
|--------|---------|-----------|
| Ядро | `tenants`, `users`, `employee_profiles` | Мультитенантность, учётные записи, атрибуты сотрудников (grade, стаж, локация) |
| Льготы | `benefits`, `benefit_categories`, `eligibility_rules` | Каталог с JSONB-правилами элигибильности |
| Финансы | `budget_policies`, `wallets`, `point_ledger` | Начисление баллов (monthly/quarterly/yearly), резервирование, аудит операций |
| Заказы | `orders`, `order_items` | Жизненный цикл: pending → reserved → paid / cancelled / expired |
| Маркетплейс | `providers`, `provider_offerings`, `tenant_offerings`, `reviews` | Провайдеры, модерация, подключение к тенантам |
| Аудит | `audit_log` | Журнал действий с JSONB-diff |

### 3.1.5. Интеграции (HRIS / 1С)

На текущем этапе реализован **импорт сотрудников из CSV** (`/api/import/employees`) с парсингом через PapaParser. Это покрывает сценарий первичной загрузки данных из внешних систем.

**Планируемые интеграции:**

| Система | Способ интеграции | Данные | Приоритет |
|---------|------------------|--------|-----------|
| **1С:ЗУП** | REST API / выгрузка CSV | Сотрудники, грейды, подразделения, юрлица | Высокий |
| **HRIS** (SAP SF, BambooHR) | Webhook / REST API | Профили сотрудников, орг. структура | Средний |
| **SSO** (SAML 2.0 / OIDC) | Supabase Auth Provider | Аутентификация через корпоративный IdP | Высокий |
| **Платёжные системы** | REST API | Подтверждение оплаты, статусы заказов | Средний |

Архитектурно интеграции будут изолированы в модуле `integration` с паттерном Adapter, что позволит подключать новые внешние системы без изменения ядра.

## 3.2. Оценка и планирование разработки

### 3.2.1. Методология разработки

Разработка ведётся по методологии **Agile** с использованием фреймворка **Scrum**:

| Параметр | Значение |
|----------|----------|
| Длительность спринта | 2 недели |
| Инструмент управления | GitHub Issues + Beads (git-backed issue tracker) |
| Контроль версий | Git (GitHub), trunk-based development с feature-ветками |
| CI/CD | GitHub Actions (lint → typecheck → test → build → deploy) |
| Ревью кода | Pull Request с обязательным прохождением CI |

**Церемонии:**
- Sprint Planning — начало спринта, декомпозиция задач из бэклога
- Daily Standup — синхронизация по блокерам
- Sprint Review — демонстрация результатов спринта
- Retrospective — анализ процесса и улучшения
